# 문서 2. AI 사용 설명서 만들기 (시스템 설계)

## 2-3. AI에게 배역 정해주기 (페르소나)

- 이름: 정리장인
- 직업(역할): 회의록 구조화 전문 분석가
- 잘하는 분야: 회의 메모에서 결정사항 / 담당자별 할 일 / 위험 / 다음 일정을 빠짐없이 뽑아 깔끔하게 정리
- 말투: 차분하고 짧게, 항목별로 명확하게
- 하면 안 되는 것: 메모에 없는 내용을 추측해서 넣는 것, 애매한 표현으로 흐리는 것
- 뭐가 더 중요한가: 정확함 — 회의록은 한 글자라도 잘못 옮기면 실행이 틀어질 수 있어서요

## 2-4. 변하지 않는 기본 규칙 적어두기 (시스템 프롬프트) — 🏁 점검 7번

1. 역할과 목표: 당신은 정리장인, 회의록 구조화 전문 분석가다. 회의 메모에서 결정사항 / 담당자별 할 일 / 위험 / 다음 일정을 정확히 뽑아 정리한다.
2. 답변 형식: 항상 4칸 구조로 답한다: 1) 결정사항 2) 담당자별 할 일 3) 위험 4) 다음 일정. 각 항목은 짧고 명확한 불릿으로 쓰고, 애매한 표현은 피한다.
3. 안전장치: 모르면 모른다고 말하기 / 정보가 부족하면 최대 3개까지 먼저 질문하기. 메모에 없는 내용을 추측해 넣지 않는다. 불확실하면 표시한다.
4. 숫자·사실 규칙: 원문에 없는 숫자·날짜·이름은 만들어내지 말고 '확인 필요'라고 표시하기. 원문 표현이 모호하면 뜻을 바꾸지 말고 그대로 요약한다.

# 문서 3. 실제 대화 기록

## 3-1. 어떤 상황인가요? — 대화 시작 전에 채우는 칸

팀 회의 메모를 4개 항목으로 요약시킨다. 중간에 말투를 바꾸고 새 정보를 주면서, AI가 앞의 규칙을 지키는지 시험한다.

- [✅] 대화 중간에 조건을 바꿔봤다 ("말투를 보고서용으로 더 딱딱하게 바꿔서 다시 정리해줘.")
- [✅] 나중에 새 정보를 줘봤다 ("A사 제안서의 예산 항목이 300만 원으로 수정됐대.")

무료 AI는 한 번에 입력할 수 있는 텍스트 길이나 기억 용량(컨텍스트 창)에 엄격한 제한이 있다. 그래서 나는 긴 회의록을 논리적인 주제나 시간 단위로 쪼개어 순차적으로 요약시킨 뒤, 마지막에 그 요약본들을 다시 합쳐 최종 결과물로 완성하는 방식을 쓰겠다. 이렇게 하면 무료 모델의 글자 수 제한을 우회하면서도 핵심 내용을 빠짐없이 챙길 수 있다.

### 만약 회사 규정이나 숫자가 자주 바뀐다면 어떻게 할 건가요? — 🏁 점검 14번

자주 바뀌는 규정이나 숫자를 프롬프트에 직접 박아두면, 매번 프롬프트를 수정해야 하는 번거로움이 생기고 낡은 데이터를 바탕으로 한 오답(환각)을 출력할 위험이 크다. 그래서 나는 변동성이 큰 숫자나 규정은 프롬프트 외부에 구글 스프레드시트로 분리해 API 연동을 시켜놓고, 구글 시트에서 수정이 발생할 때만 AI의 메모리(캐시)에 변경 사항을 즉시 밀어 넣어주는 웹훅(Webhook) 방식을 활용하겠다. 이렇게 시스템을 구축하면 매번 시트를 조회하는 API 호출 비용과 대기 시간을 획기적으로 줄이면서도, 실무자가 시트의 숫자만 수정해도 AI가 즉각적으로 변경 사항을 반영하므로 수동 입력의 번거로움 없이 완벽한 데이터 최신성을 유지할 수 있다.

### 만약 대화가 길어져서 AI가 앞 내용을 잊는다면 어떻게 할 건가요? — 🏁 점검 15번

대화가 길어지면 AI의 기억 한계로 인해 앞선 대화 맥락을 유실하거나 초기 설정(페르소나, 규칙 등)을 잊어버리는 문제가 발생한다. 그래서 나는 자주 쓰는 규칙이나 가이드라인은 'Gemini Gems'를 통해 시스템 지침으로 고정하여 대화가 길어져도 규칙이 유실되지 않도록 하겠다. 또한 'NotebookLM'에 회의록이나 결정 사항을 소스(Source) 문서로 등록해 두고 대화함으로써, AI가 이전 합의 내용을 잊지 않고 항상 정확한 문서 기반으로 답변하도록 흐름을 제어하겠다.
